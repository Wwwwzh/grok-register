# Grok Register Console 优化计划

> 状态：P0 + P1 + 号池趋势 + 暗色模式 + SSE 已实现（2026-07-16）  
> 目标：把控制台从“能用”提升到“日常运维省心”  
> 范围：`apps/console` 前端 + 必要后端 API  
> 不改：注册核心脚本逻辑、生产号池服务本身

## 1. 背景与现状

当前控制台已具备：

- 系统默认配置 / 任务级覆盖
- 任务创建、停止、删除、日志查看
- 成功 / 失败 / 入池统计与进度条
- 环境健康检查 + 号池总量（web / build / console）
- 鉴权（token cookie）与基础页面重构

仍存在的运维痛点：

1. 任务一多，查找和清理成本高
2. 失败原因埋在日志里，不够直观
3. “注册成功但未入池”不够醒目
4. 日志可读性一般，缺少下载和自动滚动控制
5. 创建任务前缺少预检，容易空跑

## 2. 目标

### 业务目标

- 3 秒内找到目标任务
- 10 秒内判断任务失败主因
- 一眼看出“成功但未入池”
- 一键清理历史终态任务
- 降低无效任务启动次数

### 体验目标

- 信息密度高但不乱
- 关键操作少跳转
- 移动端可查看，桌面端高效操作

### 非目标（本轮不做）

- 重写注册脚本
- 引入复杂权限系统
- 完整监控告警平台
- 大规模可视化 BI

## 3. 优先级与分期

### P0（本轮必做，1 个工作单元）

1. 任务搜索 + 批量清理
2. 失败摘要 + 入池失败高亮
3. 日志下载 + 自动滚动开关

### P1（下一轮）

4. 创建前预检
5. 错误类型聚合
6. 任务模板

### P2（可选）

7. 号池趋势
8. 暗色模式
9. SSE/WebSocket 推送
10. 桌面通知 / 操作审计

## 4. 详细方案

### 4.1 P0-1 任务搜索 + 批量清理

#### 前端

- 任务栏工具区新增：
  - 搜索框：支持任务 ID / 名称模糊匹配
  - 清理按钮：`清理已完成`、`清理失败/部分`、`清理全部终态`
- 清理前二次确认，展示将删除数量
- 搜索与现有筛选（全部/进行中/已完成/失败）叠加生效

#### 后端

- 新增批量删除接口，例如：
  - `POST /api/tasks/cleanup`
  - body: `{ "statuses": ["completed","stopped","failed","partial"] }`
- 返回：`deleted_count`、`deleted_ids`
- 删除语义保持现状：只删控制台任务记录与任务目录，不影响号池账号

#### 验收

- 搜索 `hk` 能筛出名称含 hk 的任务
- 一键清理后终态任务消失
- 运行中任务不会被误删

### 4.2 P0-2 失败摘要 + 入池失败高亮

#### 前端

- 任务卡新增一行 `last_error` 摘要（单行截断，hover 看全量）
- 详情摘要新增：
  - `注册成功`
  - `入池成功`
  - `入池缺口 = max(0, completed - pushed)`
- 当 `completed_count > pushed_count` 时：
  - 任务卡/详情显示红色提示：`有成功未入池`
- 状态视觉：
  - 入池缺口 > 0：橙/红色强调
  - 纯失败：红色
  - 正常完成：绿色

#### 后端（尽量复用现有字段）

- 任务列表/详情补充或确认字段：
  - `last_error`
  - `pushed_count / pushed_created / pushed_updated`
  - 可选：`push_failed_count`（若现有日志可解析）
- 若现有数据够用，可不改 schema，只做前端计算

#### 验收

- 失败任务不用点开日志也能看到主因摘要
- 成功 5、入池 3 的任务明确显示缺口 2

### 4.3 P0-3 日志下载 + 自动滚动开关

#### 前端

- 日志区增加：
  - `下载日志` 按钮
  - `自动滚动` 开关（默认开）
- 用户手动上翻时：
  - 若关闭自动滚动，刷新不打断阅读
- 下载内容：
  - 当前任务最近日志（前端导出 txt）
  - 文件名：`task-{id}-{name}.log`

#### 后端（可选增强）

- 若日志很大，再补：
  - `GET /api/tasks/{id}/logs/download`
- P0 可用前端导出先落地

#### 验收

- 能下载当前任务日志
- 关闭自动滚动后，查看历史日志不会被拽到底

## 5. P1 方案草案

### 5.1 创建前预检

- 新建任务提交前调用轻量预检：
  - 代理可达
  - temp mail 可达
  - admin login 成功
  - import endpoint 可访问
- 预检失败：
  - 阻止创建，或要求二次确认“仍要创建”

### 5.2 错误类型聚合

- 从日志/结果中归类：
  - mail
  - proxy/warp
  - captcha/turnstile
  - xai register
  - import/push
- 详情页显示计数，不只给原始 last_error

### 5.3 任务模板

- 保存常用创建参数为模板
- 一键套用名称前缀、次数、代理/邮箱覆盖

## 7. 接口草案

### 7.1 批量清理

```http
POST /api/tasks/cleanup
Content-Type: application/json

{
  "statuses": ["completed", "stopped", "failed", "partial"]
}
```

```json
{
  "deleted_count": 12,
  "deleted_ids": [3, 4, 5]
}
```

规则：

- 不允许删除 `queued/running/stopping`
- 空 statuses 返回 400

### 7.2 预检（P1）

```http
POST /api/preflight
```

```json
{
  "ok": false,
  "items": [
    {"key": "warp", "ok": true, "summary": "HTTP 200"},
    {"key": "temp_mail", "ok": false, "summary": "HTTP 403"},
    {"key": "import", "ok": true, "summary": "reachable"}
  ]
}
```

## 10. 风险与对策

| 风险 | 影响 | 对策 |
|---|---|---|
| 批量删除误伤运行中任务 | 中 | 后端强制过滤 active 状态 |
| last_error 过长撑破布局 | 低 | 单行截断 + title |
| 前端导出日志不完整 | 中 | P0 限最近 N 行；P1 加服务端下载 |
| 预检过严阻止合法任务 | 中 | 允许“预检失败仍创建”二次确认 |
| 轮询 + 新接口增加负载 | 低 | 保持当前自适应轮询 |

## 11. 完成定义（DoD）

P0 完成需同时满足：

1. 代码已推送到 `main`
2. 控制台重启后页面可用
3. 搜索 / 清理 / 失败摘要 / 入池缺口 / 日志下载 / 自动滚动 均验证通过
4. 不影响现有创建、停止、健康检查、鉴权
5. 文档与计划状态更新为“已完成”或记录偏差
